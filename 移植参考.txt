你的思路**总体是对的**，而且很务实。  
但我建议你把结论拆成“性能”和“发布/保密”两部分看：

**结论**
- 性能目标：**部分成立**，但“重写成 C++ ≠ 必然大幅提速”。
- 发布与保密目标：**成立**，C++ 在产品化交付上确实更有优势。

**性能这块怎么判断**
- 你现在这类程序的重负载通常在解码/编码（OpenCV/FFmpeg），它们本身已经是 C/C++ 实现。
- 所以很多时候瓶颈是：编码参数、I/O、缩放/叠加次数、线程模型，而不是 Python 语法本身。
- C++ 会有收益的地方主要在：
  - 每帧大量 Python 层逻辑（HUD绘制、数据插值、UI刷新）；
  - 低延迟预览路径（减少 Python/Tkinter 的主线程开销）；
  - 更细粒度并行优化。
- 导出速度想明显提升，常常先看：
  - 硬件编码（NVENC/QSV/AMF）；
  - ffmpeg preset / crf / bitrate 策略；
  - 管线并行（读帧、叠加、编码分离）。

**发布与保密这块**
- C++ 打包成 exe 的确更“像产品”，部署体验更统一。
- 但“保密”是**提高门槛**，不是绝对防破解；C++ 也可被逆向。
- 相比纯 Python 分发，C++ 在代码暴露上通常更少，商业上更友好。
- Python 也可做增强方案（PyInstaller/Nuitka/Cython + 关键逻辑 native 化）。

**我给你的建议路线（最稳）**
- 先做一次性能剖析（预览 FPS、导出 fps、CPU/GPU 占用、I/O占比）。
- 先优化现有 Python+FFmpeg 管线，拿到“低成本收益”。
- 再做“混合架构”：
  - UI/流程仍可先保留 Python；
  - 把核心热点（HUD渲染、插值、批处理）迁到 C++/pybind11；
  - 最后视收益决定是否全量迁移 Qt/C++。
